1Architecture

This deployment uses the pre-provisioned provider — NKP assumes the OS is already installed on each node. A temporary KIND bootstrap cluster on the first node acts as the CAPI management plane, provisions the target cluster, then pivots (self-manages) into it. All container images come from a local Docker registry on 192.168.1.159:5000 — no internet access is required or attempted after initial bundle extraction.

2Official Docs vs Reality

The Nutanix NKP 2.17 documentation describes a cloud-provisioned happy path. Pre-provisioned air-gapped bare metal diverges significantly. This table captures every major gap discovered across 7 deploy sessions.

Topic What the Docs Say What You Actually Have to Do Gotcha
Bundle structure "Use --bundle flag with the downloaded archive" Extract the 22 GB .tar.gz first. --bundle wants inner .tar files, not the outer archive. G-03, G-11
Bootstrap registry "Bootstrap cluster includes an internal registry" The bootstrap internal registry has no NodePort — external nodes cannot reach it. You must run a separate Docker registry:2 container and push with nkp push bundle. G-04, G-18
SSH user Default SSH username is "konvoy" Pass --ssh-username ice (or your OS user). "konvoy" will fail because the user doesn't exist. G-05
SSH auth wiring "Pass --ssh-private-key-file to configure SSH access" The flag creates a Kubernetes Secret but does not patch it into PreprovisionedInventory.spec.sshConfig. You must manually patch both inventories after cluster create. G-22
PreprovisionedInventory API group Docs show CAPI's cluster.x-k8s.io group NKP uses its own infrastructure.cluster.konvoy.d2iq.io/v1alpha1 API group. The upstream group fails with "no matches for kind." G-20
Workers required Docs assume at least one worker nodepool Even with --worker-replicas 0, you must include a worker PreprovisionedInventory with hosts: [] or the command errors. G-20
kube-vip / VIP Docs say "provide a VIP address" — no further setup described kube-vip is NOT auto-deployed in pre-provisioned mode. You must either: add the VIP IP to the interface manually + run kubeadm phases, or pre-deploy kube-vip as a static pod. G-29
cert-manager Docs say "Kommander installs cert-manager automatically" Pre-provisioned mode skips cert-manager ClusterResourceSet. Kommander preflight checks for it and hard-stops. Install cert-manager via Helm from local registry first. G-31
Kommander airgapped flag "Use --airgapped to skip external connections" Even with --airgapped, the installer tries to fetch kommander-applications from GitHub. Pass --kommander-applications-repository /local/path/...tar.gz with a local path. G-32
flux-oci-mirror Docs describe flux-oci-mirror as the air-gap OCI proxy flux-oci-mirror requires MITM TLS certs auto-generated during cloud provisioning. In pre-provisioned mode the certs are missing — proxy is completely non-functional. Bypass it by patching OCIRepository URLs to point directly to local registry. G-35
OCIRepository URLs Flux resources should auto-configure for air-gap All OCIRepositories in the git-operator repo hardcode ghcr.io and docker.io. You must clone the internal git repo, replace all external refs with 192.168.1.159:5000, and push a new commit. G-37, G-40
kommander-operator OCIRepositories Operators respect the airgapped configuration kommander-operator hardcodes ghcr.io in two OCIRepositories it owns and will revert any manual patches. Must scale operator to 0, patch, wait for Ready=True, scale back. G-36, G-42
local-volume-provisioner "Create directories for local storage" Plain mkdir directories are rejected. Provisioner validates against /proc/mounts. Must use mount --bind /path /path to create real mount points. G-33
containerd runtime "Ensure containerd is installed" NKP bundles its own containerd (v1.7.29-d2iq). The NIB Ansible removes the stock containerd.io package — on the bootstrap node this kills Docker and the KIND cluster. Must neuter dpkg prerm scripts and enable Docker live-restore first. G-27, G-28
RUNTIME_NAMESPACE env var Source-controller auto-configures via Downward API In pre-provisioned mode the RUNTIME_NAMESPACE fieldRef is absent from source-controller. Artifact URLs contain literal $(RUNTIME_NAMESPACE) — kustomize-controller fails DNS lookup. Must set manually until kommander-flux HelmRelease applies the chart correctly. G-34, G-38

Phase 1 Node Prerequisites SCRIPTED

Run on all 3 nodes before any NKP operations:

# Disable swap (kubelet refuses to start with swap enabled)
swapoff -a
sed -i -E '/^[^#].*[[:space:]]swap[[:space:]]/s/^/#/' /etc/fstab

# Add passwordless sudo for deploy user
echo 'ice ALL=(ALL) NOPASSWD:ALL' | sudo tee /etc/sudoers.d/nkp-ice
chmod 440 /etc/sudoers.d/nkp-ice

# Import Kubernetes apt repo GPG key (NIB Ansible needs this — G-24)
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.30/deb/Release.key \
  | sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

# Create bind-mount volumes for local-volume-provisioner (G-33)
for i in 01 02 03; do
  sudo mkdir -p /mnt/disks/vol-$i
  sudo mount --bind /mnt/disks/vol-$i /mnt/disks/vol-$i
done
G-27/G-28 CRITICAL (bootstrap node 159 only): Before running cluster create, neuter the containerd.io dpkg prerm scripts and enable Docker live-restore. Otherwise NIB's apt-get remove containerd.io will kill the KIND cluster. 
# On 159 ONLY — neuter prerm scripts
for PKG in containerd.io docker-ce; do
  PRERM="/var/lib/dpkg/info/${PKG}.prerm"
  [ -f "$PRERM" ] && sudo cp "$PRERM" "${PRERM}.bak" && \
    printf '#!/bin/sh\nexit 0\n' | sudo tee "$PRERM" > /dev/null
done
# Pre-remove containerd.io to avoid EBUSY systemd conflict (G-28)
sudo DEBIAN_FRONTEND=noninteractive apt-get remove -y containerd.io
sudo systemctl daemon-reload
sudo pkill -TERM -x containerd 2>/dev/null || true
# Enable live-restore
echo '{"live-restore": true}' | sudo tee /etc/docker/daemon.json
sudo systemctl reload-or-restart docker

Phase 2 — Registry Setup

Phase 2 Local Docker Registry + containerd Mirror Config SCRIPTED 
# Start Docker registry on 159 (if not already running)
docker run -d --name registry --restart=always \
  -p 5000:5000 -v /var/lib/registry:/var/lib/registry \
  registry:2

# Configure containerd mirror on ALL 3 nodes (catch-all redirect)
# This sends ALL image pulls through the local registry
for NODE in 192.168.1.159 192.168.1.158 192.168.1.156; do
  ssh ice@$NODE "sudo mkdir -p /etc/containerd/certs.d/_default && sudo tee /etc/containerd/certs.d/_default/hosts.toml" <<'EOF'
[host."http://192.168.1.159:5000"]
  capabilities = ["pull", "resolve"]
  skip_verify = true
EOF
done
Format note (G-04): REGISTRY_IP=192.168.1.159:5000 (no scheme) is used for nkp push bundle --to-registry. REGISTRY_MIRROR_URL=https://192.168.1.159:5000 (with scheme) is used for nkp create cluster --registry-mirror-url. These are different flags with different format requirements.

Phase 3 — Push Bundles

Phase 3 Push Konvoy + Kommander Bundles to Registry SCRIPTED 
BUNDLE_DIR="$HOME/nkp-bundle/nkp-v2.17.1/container-images"
REGISTRY_IP="192.168.1.159:5000"

# Push Konvoy bundle (~11 GB)
nkp push bundle \
  --bundle "$BUNDLE_DIR/konvoy-image-bundle-v2.17.1.tar" \
  --to-registry "$REGISTRY_IP" \
  --to-registry-insecure-skip-tls-verify \
  --image-push-concurrency 4

# Push Kommander bundle (~11 GB) — needed for cert-manager charts (G-31)
nkp push bundle \
  --bundle "$BUNDLE_DIR/kommander-image-bundle-v2.17.1.tar" \
  --to-registry "$REGISTRY_IP" \
  --to-registry-insecure-skip-tls-verify \
  --image-push-concurrency 4
G-19: nkp push bundle uses an ephemeral internal staging registry on a random port. If it dies mid-push, the command exits 0 but images may be missing. Re-run once more — it is idempotent and skips already-pushed images.

Phase 4 — Bootstrap KIND Cluster

Phase 4 Create KIND Bootstrap Cluster SCRIPTED 
BUNDLE_DIR="$HOME/nkp-bundle/nkp-v2.17.1"
REGISTRY_MIRROR_URL="https://192.168.1.159:5000"

# Bootstrap loads images from the extracted .tar files into KIND
# Do NOT use --bundle here (G-18: creates inaccessible internal registry)
nkp create bootstrap \
  --bootstrap-cluster-image "$BUNDLE_DIR/konvoy-bootstrap-image-v2.17.1.tar" \
  --registry-mirror-url "$REGISTRY_MIRROR_URL" \
  --registry-mirror-insecure \
  --registry-insecure-skip-tls-verify \
  --kubeconfig ~/.kube/nkp-bootstrap.kubeconfig
G-21: After bootstrap, if the KIND container restarts, the API port may change (e.g., 40047 → 38077). Extract a fresh kubeconfig: docker exec konvoy-capi-bootstrapper-control-plane cat /etc/kubernetes/admin.conf | sed 's|https://.*:6443|https://127.0.0.1:NEW_PORT|g' > ~/.kube/nkp-bootstrap.kubeconfig
G-26: After a KIND container restart, wait 2–3 minutes for the API server to become ready before running kubectl commands. Poll: curl -sk https://127.0.0.1:38077/healthz

  --control-plane-endpoint-port 6443 \

  --timeout 2h \

# CRITICAL: Patch SSH config into PreprovisionedInventory (G-22)
for inv in nkp-cluster-control-plane nkp-cluster-md-0; do
  "$KC" $KCF patch preprovisionedinventory "$inv" -n default --type=merge \
    -p '{"spec":{"sshConfig":{"port":22,"user":"ice","privateKeyRef":{"name":"nkp-cluster-ssh-key","namespace":"default"}}}}'
done

# CRITICAL: Add NKP deploy key to authorized_keys on all nodes (G-23)
NKP_PUBKEY=$("$KC" $KCF get secret nkp-cluster-ssh-key -n default \
  -o jsonpath='{.data.ssh-privatekey}' | base64 -d | ssh-keygen -y -f /dev/stdin)
echo "$NKP_PUBKEY nkp-deploy" >> ~/.ssh/authorized_keys
for NODE in 192.168.1.158 192.168.1.156; do
  sshpass -p 'PASSWORD' ssh -o StrictHostKeyChecking=no ice@$NODE \
    "echo '$NKP_PUBKEY nkp-deploy' >> ~/.ssh/authorized_keys"
done

sudo ip addr add 192.168.1.100/24 dev ens33
sudo kubeadm init phase upload-certs --upload-certs --config /run/kubeadm/kubeadm.yaml
sudo kubeadm init phase upload-config all --config /run/kubeadm/kubeadm.yaml
sudo kubeadm init phase mark-control-plane --config /run/kubeadm/kubeadm.yaml
sudo kubeadm init phase bootstrap-token --config /run/kubeadm/kubeadm.yaml
sudo kubeadm init phase addon all --config /run/kubeadm/kubeadm.yaml
sudo mkdir -p /run/cluster-api
echo "success" | sudo tee /run/cluster-api/bootstrap-success.complete

# Fix iptables after kube-proxy sets FORWARD=DROP (G-30)
KIND_BR="br-$(docker network ls --filter name=kind --format '{{.ID}}' | cut -c1-12)"
sudo iptables -t nat -A POSTROUTING -s 172.17.0.0/16 ! -o docker0 -j MASQUERADE
sudo iptables -t nat -A POSTROUTING -s 172.18.0.0/16 -j MASQUERADE
sudo iptables -I FORWARD 1 -i "$KIND_BR" -j ACCEPT
sudo iptables -I FORWARD 1 -o "$KIND_BR" -j ACCEPT

4Flux & Kommander Internals

Understanding who owns what is critical for debugging. Two different entities manage OCIRepository resources — confusing them leads to chasing symptoms instead of root causes.

OCIRepository Owner Source URL Air-Gap Fix
platform-version-v2-17-1 kommander-operator (hardcoded) oci://ghcr.io/mesosphere/... Race: scale op→0, patch URL, wait Ready=True, scale op→1
kommander-appmanagement-0.17.1-chart kommander-operator (via git repo) oci://docker.io/mesosphere/... Fix docker.io ref in git repo (G-40) + scale-0 race
Per-app chart repos (dex, traefik, etc.) kommander-appmanagement Already uses local registry (from apps tarball) No action needed — auto-correct via apps tarball
All other resources in git repo Kustomization (Flux GitRepository) oci://ghcr.io/... (original) sed replacement in git repo commit (G-37)

The OCI URL Fix Sequence

K="~/nkp-bundle/nkp-v2.17.1/kubectl"
KC="--kubeconfig ~/.kube/nkp-cluster.kubeconfig"

# Step 1: Get git-operator credentials and clone
GIT_PASS=$($K $KC get secret kommander-git-credentials -n kommander-flux \
  -o jsonpath='{.data.password}' | base64 -d)
GIT_IP=$($K $KC get svc git-operator-git -n git-operator-system \
  -o jsonpath='{.spec.clusterIP}')
GIT_URL="https://kommander:${GIT_PASS}@${GIT_IP}/repositories/kommander/kommander.git"
GIT_SSL_NO_VERIFY=true git clone "$GIT_URL" /tmp/kommander-git
cd /tmp/kommander-git

# Step 2: Replace ALL external registry refs (both ghcr.io AND docker.io)
find . -name "*.yaml" | xargs sed -i \
  's|oci://ghcr.io/|oci://192.168.1.159:5000/|g;
   s|image: ghcr.io/|image: 192.168.1.159:5000/|g;
   s|oci://docker.io/|oci://192.168.1.159:5000/|g'

# Step 3: Remove catalogCollections with port numbers (G-39)
# Edit applications/kommander/0.17.1/helmrelease/cm.yaml manually
# or patch the ConfigMap after Kommander installs

git add -A && git commit -m "fix: redirect all OCI refs to local registry"
GIT_SSL_NO_VERIFY=true git pull --rebase
GIT_SSL_NO_VERIFY=true git push

# Step 4: Fix platform-version-v2-17-1 (operator-owned, not in git repo)
$K $KC scale deployment kommander-operator -n kommander --replicas=0
$K $KC patch ocirepository platform-version-v2-17-1 -n kommander --type=merge \
  -p '{"spec":{"url":"oci://192.168.1.159:5000/mesosphere/kommander-applications/collection","insecure":true,"ref":{"tag":"v2.17.1"},"suspend":false}}'
sleep 15
$K $KC scale deployment kommander-operator -n kommander --replicas=1

6Dashboard Access

The Kommander dashboard at https://192.168.1.200/dkp/kommander/dashboard is on a private LAN. Accessing it from a Windows workstation requires an SSH tunnel + Python reverse proxy to handle the Dex OIDC authentication flow (which involves HTTP redirects to 192.168.1.200).

Setup

# Step 1: SSH tunnel on Windows (PowerShell or terminal)
ssh -L 8888:192.168.1.200:443 -N ice@192.168.1.159

# Step 2: Start Python reverse proxy on 159
# The proxy rewrites Location headers and strips cookie domain/secure attributes
#!/usr/bin/env python3
# nkp-proxy.py — run on 192.168.1.159
import http.server, urllib.request, urllib.error, ssl, re, socket, sys

TARGET = "https://192.168.1.200"
LOCAL  = "http://localhost:8888"
PORT   = 8888

class ProxyHandler(http.server.BaseHTTPRequestHandler):
    def do_request(self, method):
        url = TARGET + self.path
        body = None
        if method in ("POST", "PUT", "PATCH"):
            length = int(self.headers.get("Content-Length", 0))
            body = self.rfile.read(length) if length else None
        headers = {k: v for k, v in self.headers.items()
                   if k.lower() not in ("host", "connection", "transfer-encoding")}
        headers["Host"] = "192.168.1.200"
        req = urllib.request.Request(url, data=body, headers=headers, method=method)
        ctx = ssl.create_default_context()
        ctx.check_hostname = False
        ctx.verify_mode = ssl.CERT_NONE
        try:
            with urllib.request.urlopen(req, context=ctx) as resp:
                self.send_response(resp.status)
                for k, v in resp.headers.items():
                    if k.lower() == "location":
                        v = v.replace(TARGET, LOCAL)
                    if k.lower() == "set-cookie":
                        v = re.sub(r';\s*[Ss]ecure\b', '', v)
                        v = re.sub(r';\s*[Ss]ame[Ss]ite=[A-Za-z]+', '', v)
                        v = re.sub(r';\s*[Dd]omain=[^;,\s]+', '', v)
                    if k.lower() not in ("transfer-encoding",):
                        self.send_header(k, v)
                self.end_headers()
                self.wfile.write(resp.read())
        except urllib.error.HTTPError as e:
            self.send_response(e.code)
            for k, v in e.headers.items():
                if k.lower() == "location":
                    v = v.replace(TARGET, LOCAL)
                if k.lower() not in ("transfer-encoding",):
                    self.send_header(k, v)
            self.end_headers()
        except Exception as ex:
            self.send_error(502, str(ex))
    def log_message(self, fmt, *args): pass
    do_GET = do_POST = do_PUT = do_DELETE = do_PATCH = do_HEAD = do_request

httpd = http.server.HTTPServer(("127.0.0.1", PORT), ProxyHandler)
print(f"Proxy running on {LOCAL}", flush=True)
httpd.serve_forever()
# Run proxy on 159 (in background or tmux)
python3 /tmp/nkp-proxy.py &

Why strip Domain= from cookies? The cookie is set with Domain=192.168.1.200. When the browser receives it from localhost:8888, the domain mismatch causes the cookie to be silently discarded — resulting in "Not Authorized" on every page despite successful login. Stripping the Domain attribute lets the browser accept and send the cookie for any origin.
